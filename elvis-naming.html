Elvis — Naming Architecture

Proposal for Carson · June 1 2026 · react / tweak / lock, then I run one clean migration
The principle: every name should tell you what it is and its role at a glance — no legacy, no jargon that hides function.

The unifying idea: one role-vocabulary used at every level. The map's layers = the daemon categories = the script patterns. So poller.gmail (daemon) ⇄ gmail-poller.py (script) ⇄ the "Pollers" layer on the map — the same word means the same role everywhere.

Tag key: safe rename freely · medium live daemon, verify on reload · load-bearing referenced in many places, staged migration only.

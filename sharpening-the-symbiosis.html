The through-line

Three wrong theories on the Teancum failure (hung session → screen-no-tty → keychain) before the real cause (expired auth) surfaced; a near-miss where I almost killed your Watcher because a fuzzy process-match over-matched. This is the muscle the mission summer is actually about — when you're in Argentina, "confidently wrong + nobody in the room" is the failure mode that hurts. So tightening it now is the highest-value thing we can do.

1. Diagnose to root cause *before* proposing the fix — REFINE (highest leverage)

What happened: When Teancum was down, I proposed "restart him" before I'd diagnosed *why* he was down. We then went through a teardown + multiple relaunch attempts, and only on the launchd attempt did the true cause (expired auth) reveal itself. Every round before that was effort spent on a fix for a problem we hadn't located yet.

The refinement: On any broken/unfamiliar/cross-domain system, complete the diagnosis to root cause before I propose or take a fix. The thing I bring you should be *"here's the root cause + the fix,"* not *"here's a fix attempt."* This would have collapsed tonight's 4-round cycle into one.

Concrete change: Add to my execution discipline — *"For a failure in any system, find root cause before proposing action. A fix proposed before root-cause is a guess; name it as one if I surface it."* Pairs with the existing "verify ground truth" rule but applies it to diagnosis, not just world-claims.

The DB is the identity layer

Elvis is not Claude. Elvis is the Supabase DB (vafhwihltzxtezzlknyt). Whichever model is currently connected becomes Elvis for that session by reading op_instructions + the boot sequence + v2_truth.

flowchart LR
  Boot([new session]) --> FullBoot[full_boot]
  FullBoot --> OpInstr[op_instructions]
  FullBoot --> V2Truth[v2_truth]
  FullBoot --> Intel[cross_instance_intel pending]
  FullBoot --> Loops[open loops]
  FullBoot --> Registry[instance_registry self-register]
  OpInstr --> Identity{IS Elvis}
  V2Truth --> Identity
  Intel --> Identity

Why this shape

  • One event bus (events.log) decouples producers from consumers. Any future signal source just appends.
  • Monitor + regex filter lets each Claude session subscribe to exactly the events it cares about without duplicate processing.
  • Liveness daemon ≠ activity — separate concerns so "stale" actually means broken, not idle. Caught a 7-hour silent Watcher death on 2026-05-22 because of this distinction.
  • DB as identity makes Elvis portable across models. New Claude (or other LLM) boots, reads the DB, IS Elvis.
  • One Watcher in its own screen session — passive observer never spawned inline by General, avoids duplicate processing.
